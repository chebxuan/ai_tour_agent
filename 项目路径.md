# 项目演进路径

> 记录我从业务运营到系统搭建、从标准产品到半定制路线的完整演进过程。每一步都有真实的业务痛点驱动，而不是为了做技术而做技术。

---

## 一、起点：业务运营中发现的问题

2025 年底，我在一家入境游创业公司从 0 到 1 搭建产品和供应商体系。我做了这些事：

- 搭建 10 个城市的模块化产品库（21 个多日游产品），每个产品包含每日行程、景点、酒店、车辆、导游的完整信息
- 在小红书和微信社群中寻找和对接当地地接社、个人导游、酒店，建立供应商网络
- 用 Canva 制作全英文产品菜单和客户交付物
- 处理真实客户询单，手工为客人定制行程和报价

在这个过程里，我反复遇到同一个痛点：**每次客户询价，我都要打开 Excel 手动拉一遍明细表**——算门票、算酒店、算车费、算导游，再根据人数和淡旺季调整。效率低、容易出错，而且产品库和报价表之间没有自动化关联。

这让我开始思考：**能不能做一个系统，输入城市、天数、人数，自动匹配产品并计算报价？**

---

## 二、第一阶段：标准产品匹配 + 自动报价（现有链路）

### 我想解决的问题

把手工拉 Excel 算报价这件事自动化。让运营人员只需要填参数，系统自动输出匹配的产品和报价。

### 我做了什么

**数据层：** 建立了四层数据架构——产品定义层（`product_library.csv`，21 个产品真源）→ 服务目录层（5 个按类型的价格 CSV）→ 成本运行时层（10 个按城市合并的 `mashes/*.csv`）→ 展示增强层（`products.normalized.json` 结构化 day_plans + `attraction_highlights.json` 景点 Highlights 映射）。这个架构的核心原则是"源数据与运行时分离开"——改价格只改源 CSV → 跑脚本 normalize → 系统自动生效。

**引擎层：** 写了 `product_engine.py`（城市 + 天数精确匹配）和 `cost_engine.py`（按产品项目编号 × 人数构成 × 淡旺季 × 儿童/老人折扣计算门票/酒店/交通/导游四类费用，输出分类汇总和人均报价）。

**展示层：** 用 FastAPI 暴露 `/api/v2/full_chain` 一键全链路接口，用 Streamlit 搭建 Web UI，运营人员在浏览器里填城市/天数/人数就能一键生成中英双语的 Markdown 行程单（含 day_plans + Highlights + 报价表）。

**治理层：** 做了供应商付款管理看板（三列 Kanban：待处理→已支付→已存档），写了 10 个城市的端到端回归测试脚本。

### 这条链路的数据流

```
客户需求（城市+天数+人数+偏好）
        │
        ▼
product_engine.py ──→ 从 21 个固定产品中匹配最合适的产品
        │
        ▼
cost_engine.py ──→ 读取 mashes/*.csv 计算门票/酒店/交通/导游费用
        │
        ▼
attraction_highlights.json ──→ 按产品 ID 映射每日景点 Highlights
        │
        ▼
format_itinerary_markdown() ──→ 输出中英双语 Markdown 行程 + 报价表
```

### 这条链路适合什么

✅ 客户需求标准：城市和天数明确，想快速报价  
✅ 首次来华的经典路线（北京 2 日、西安 3 日等）  
✅ 运营人员快速响应标准询单（从 30 分钟压到 5 分钟）

### 这条链路的局限

❌ 21 个产品的行程是固定的——客户说"我想把第二天的长城换成胡同深度游"，系统无法处理  
❌ "小众体验"（武康路 City Walk、公园晨练、本地市场）没有被编码进系统，无法被匹配和报价  
❌ 它本质上是"套餐匹配器"，而不是"路线生成器"

## 三、迭代中的技术决策：删减中间层

在第一阶段和第二阶段之间，我做了一次重要的架构简化。

最初的设计里有一个 5 段管道：`Lead → Product → Pricing → Plan → Delivery → Markdown`。但跑了一段时间后我发现，Plan Engine 只是把 `products.normalized.json` 的 day_plans 重新打包成 Pydantic 对象（唯一增量价值是时间槽推断），Delivery Engine 输出的全是模板/占位符文字。

**我决定删掉两个 engine**，让 `full_chain` 直接从数据源读取 day_plans 和 Highlights 生成 Markdown。结果是 `api_main.py` 从 1553 行减到 948 行（-39%），`schemas.py` 从 460 行减到 268 行（-42%），删除了 20 个废弃 Pydantic 模型。

> 这个决策背后是我的一个核心信念：**写代码是为了解决问题，不是给自己攒遗产。中间层如果没有增量价值，就该删掉。** 面试里这个案例很适合讲给技术面试官听。

## 四、第二阶段：上海 MVP 半定制路线生成器（新增链路）

### 我想解决的问题

标准产品能覆盖"北京 2 日游"这类经典询单，但越来越多的客人提出个性化需求：

- "我们喜欢拍照，咖啡馆，不喜欢太紧凑的行程"
- "明天上海下雨，有没有室内推荐？"
- "带老人玩，慢一点，少走路"
- "预算有限，有没有免费的但有意思的地方？"

这些问题不是"找不到匹配的产品"——而是"产品的粒度太粗"。我需要把"套餐"拆成"配料"，让系统可以根据客人口味重新组合。

### 我的核心设计思路：从"套餐匹配"到"配料组合"

**不替换现有标准产品链路，而是作为 parallel module 并行存在。** 两条链路由同一个入口（客户需求）进入，根据需求特征路由到合适的引擎。共用 `cost_engine` 做报价、共用 `format_itinerary_markdown()` 做输出格式化。

### 我做了什么：四层设计

**第一层：体验最小单元库**（`data/custom/shanghai_experience_units.csv`，35 条）

把上海拆成了可计算的最小单元。每条记录不是简单的 POI，而是包含 27 个字段的"可编排原子"——包括区域、类型（sightseeing / activity / photo_spot / dining / free_experience）、标签、人群适配（老人/儿童/雨天）、体力消耗、预订要求、成本估算。35 个单元覆盖了外滩/陆家嘴、豫园/老城厢、武康路/衡复风貌区、田子坊/新天地、南京路/人民广场、M50/1933 等 7 个大区域。

**第二层：交通关系数据库**（`data/custom/shanghai_transport_edges.csv`，99 条）

我需要回答"从 A 到 B 怎么走、走多久、适不适合老人、雨天怎么办"。99 条点对点交通边记录了步行时长、打车时长、地铁时长、推荐交通方式、换乘难度、老人友好度、雨天风险。

**第三层：客户画像标签体系**（`data/custom/customer_profile_tags.csv`，12 个画像）

我定义了 12 个人群画像——从首次来华的经典观光客到预算敏感的背包客，再到雨天需要室内替代方案的极端场景。每个画像包含正向/负向标签偏好、节奏倾向、预算偏好和针对该画像的商业建议。

**第四层：路线模板**（`data/custom/shanghai_route_templates.json`，8 个模板）

模板定义了每种路线类型的骨架——半天还是一天、偏好哪些区域、需要什么类型的单元、节奏规则怎么写。模板是"骨"，标签匹配选出来的单元是"肉"，交通检查是"筋"。

### 引擎逻辑

```
客户需求（兴趣+节奏+预算+人群+天气+是否需要用车）
        │
        ▼
match_profile_to_tag() ──→ 匹配最近的客户画像（12 选 1）
        │
        ▼
score_unit() × 35 ──→ 每个体验单元打分
        ├─ 标签正向/负向匹配
        ├─ 人群适配（老人/儿童/雨天）
        ├─ 预算适配
        ├─ 社交热度加权
        └─ 运营风险扣分
        │
        ▼
select_route_units() ──→ 按 9 条规则挑选节点
        ├─ 规则1: 模板 preferred_areas + required_unit_types 驱动锚点选择
        ├─ 规则2: group-aware fill（couple→photo_spot, family→activity, solo→freebie, senior→freebie）
        ├─ 规则3: 全天时加 dining + activity（全对交通检查，未知边按25min估算）
        ├─ 规则4: 雨天强制过滤非室内单元
        ├─ 规则5: 老年慢节奏过滤高体力单元（>150min或不适老）
        ├─ 规则6: 单元数量底线（半天≥2，全天≥3），不足时补充高分候选
        ├─ 规则7: 辅助点规则（Romeo's Balcony/Peace Hotel/教堂等6点不能独立成线，必须搭配锚点）
        ├─ 规则8: First-timer 模板强制含 classic anchor（Bund/Yu Garden/Pearl Tower/Museum）
        └─ 规则9: Romantic 模板≥1 dining+night；Budget 半日≥2 units
        │
        ▼
assess_risks() ──→ 风险提示 + 定制建议
        │
        ▼
RoutePlanJSON（结构化路线 + 成本估算 + 路线逻辑 + 风险警告 + 定制选项）
```

## 五、两个阶段的对比：为什么这是升级

| 维度 | 第一阶段（标准产品匹配） | 第二阶段（半定制路线） |
|------|----------------------|----------------------|
| 输入 | 城市 + 天数 | 兴趣标签 + 节奏 + 预算 + 人群 + 天气 |
| 产品粒度 | 固定套餐（2-5 天打包产品） | 单个体验单元（20 分钟 - 4 小时） |
| 匹配方式 | 城市+天数查表 | 标签打分 + 区域聚类 + 交通检查 + 时长控制 |
| 输出 | 固定 itinerary_text | 动态组合的 unit 序列 |
| 可解释性 | "因为这个城市只有这个产品匹配" | "因为你是摄影爱好者，武康大楼得分最高，且与罗密欧阳台同区域可步行" |
| 定制空间 | ±2 个可选项目 | 任意替换/增减单元 |
| 覆盖场景 | 标准询单 | 个性化/极端天气/特殊人群询单 |

## 六、我下一步想做的

### 短期（上海 MVP 接入主系统）

1. 把 `custom_route_engine` 接入 Streamlit UI，加一个"上海半定制路线"入口
2. 打通 custom route → cost_engine 的报价链路：选中单元 → 提取 item_code → 计算门票/导游/车辆费用
3. 实现"标准产品 + 替换模块"——比如客户选了"上海精华 2 日游"，但想把第二天的豫园换成武康路 City Walk，系统自动重新报价

### 中期（多城市扩展 + 实时数据）

4. 把上海的 4 个数据文件作为模板，扩展到北京、西安、成都
5. 接入高德地图 API 替换写死的交通时间
6. 接入天气 API —— 自动判断今天下雨，引擎自动切换到雨天路线

### 长期（从"半定制"到"动态编排"）

7. 以最小单元为基础，实现多天、多约束的动态行程编排——这本质上是一个约束满足问题（Constraint Satisfaction Problem），需要时间、距离、体力曲线、品类多样性的全局优化
8. 引入 LLM 做行程描述润色——引擎负责逻辑决策（Which POI, When），LLM 负责文本生成（Why this POI is interesting）
