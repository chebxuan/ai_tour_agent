4. 按标准流程反馈数据问题给技术同学。

## 3. 你需要知道的核心概念

### 3.1 五个核心对象

1. `LeadJSON`：客户线索对象
2. `CandidateProductsJSON`：候选产品对象
3. `PricingResultJSON`：报价事实对象
4. `PlanObject`：行程骨架对象
5. `DeliveryDraftObject`：交付草稿对象

### 3.2 数据真源（运营必须遵守）

- 产品真源：`data/products/product_library.csv`
- 成本真源：`mashes/*.csv`

规则：

1. 先修真源数据，再重跑验证；
2. 不通过手工“临时解释”绕过结构化问题；
3. 任何价格解释以 `PricingResultJSON` 为准。

## 4. 日常运营工作流（标准 SOP）

### 步骤 1：收集线索并形成 LeadJSON

必要信息：

- 目的地城市（可多城市）
- 出行天数
- 人数结构（成人/儿童/老人）
- 时间窗（可选但建议）
- 兴趣偏好（history/culture/food 等）

输出标准：

- 字段完整优先于“文案漂亮”；
- 模糊描述可放 notes 字段，不要丢失。

### 步骤 2：产品匹配

调用 `POST /api/v2/product-match`，拿到：

- 候选产品列表（含匹配分）
- 推荐产品 ID
- 匹配原因

运营动作：

1. 看 `recommended_product_id`；
2. 检查候选是否与客户城市/天数匹配；
3. 不直接用自然语言替代结构化候选结果。

### 步骤 3：报价

调用 `POST /api/v2/pricing`，输入 `lead_id + selected_product_id`。

重点字段：

- `summary.grand_total`
- `line_items`
- `validation_issues`

运营动作：

1. 先看 `success`；
2. 有 `validation_issues` 时必须标注风险后再对客；
3. 可选项不默认并入 base scope。

### 步骤 4：行程骨架

调用 `POST /api/v2/plan`，输入 `selected_product_ids`（按拼接顺序）。

重点字段：

- `day_plans`
- `planning_notes`
- `selected_products`

运营动作：

1. 多产品时先确认产品顺序；
2. 缺失信息记录在 `planning_notes`；
3. 不把 optional 默认为 included。

### 步骤 5：交付草稿

调用 `POST /api/v2/delivery`。

重点字段：

- `sections`
- `trip_summary`
- `global_reminders`

运营动作：

1. 在结构化草稿上补联系人、执行细节；
2. 缺失项标“待确认”，不要编造；
3. 输出前做一次“价格-行程-交付一致性复核”。

## 5. 运营常用调用模板

```bash
uvicorn api_main:app --host 0.0.0.0 --port 8000 --reload
```

  -H "Content-Type: application/json" \

## 6. 运营验收清单（上线前必过）

每个单子至少检查以下 8 项：

1. 城市与天数是否与客户确认一致；
2. 推荐产品是否在候选列表中；
3. 报价是否 `success=true`；
4. `validation_issues` 是否为空或已解释；
5. optional 项是否被错误计入 base scope；
6. Plan 天数是否连续；
7. Delivery section 是否完整；
8. 对客版本与内部结构化对象是否一致。

## 7. 常见问题与处理方法

### Q1：产品匹配为空

先检查：

1. 城市是否在支持列表；
2. 天数是否有对应产品；
3. lead 字段是否为空。

处理：

- 补齐 lead 再调接口；
- 若仍为空，提交数据缺口给产品库维护人。

### Q2：报价失败或价格异常

先看 `validation_issues` 与 `error` 字段。

处理：

1. 运行 `python verify_product_cost_mapping.py`；
2. 确认产品 code 是否都能在 `mashes/{city}_merged.csv` 找到；
3. 免费项、空价项按规则标注，不直接强行补价。

### Q3：Plan 与 Delivery 信息不一致

处理：

1. 先确认 `selected_product_ids` 顺序一致；
2. 检查 `planning_notes` 是否有未承接项；
3. 重新生成 delivery，并对比 sections。

### Q4：接口 401

原因：

- `X-API-Key` 不正确或未传。

处理：

- 使用正确 header：`X-API-Key: hexa-tour-2024`（以环境配置为准）。

## 8. 数据治理协作规范（运营->技术）

提 bug 时请提供：

1. `lead_id`
2. 接口路径
3. 请求体（脱敏）
4. 返回结果（完整 JSON）
5. 预期 vs 实际
6. 是否可稳定复现

优先级建议：

- P0：报价不可用、主链路中断
- P1：字段缺失、可手工兜底
- P2：文案/展示优化建议
