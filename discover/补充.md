#### （二）核心问答环节（约50分钟）

**Mentor：当你在过往工作中负责的AI功能需要接入或优化一个新模型时，你的完整评估和决策流程是怎样的？你最关心哪些指标？**
> 我：我的流程是一个“定义问题-权衡方案-数据验证”的闭环。比如，我曾负责优化一个文本生成功能的“风格可控性”。我们和算法同学一起列出了从优化提示词、做轻微微调到更换模型等几种路径。评估时，我们会从预期效果、研发与计算成本、时间周期和长期可维护性四个维度画一个简单的决策矩阵。我最关心的指标分两层：一是模型层的直接表现，如风格匹配度、响应速度；二是能直接体现用户价值的产品层指标，比如生成内容的“用户直接采纳率”。最终我们会选择一个最匹配当前产品阶段和资源约束的方案，并通过严格的A/B测试验证后才全量。

**Mentor：那么功能上线后，如何构建数据闭环来驱动持续优化？举一个你实际处理的例子。**
> 我：以我参与过的一个信息摘要功能为例。上线后我们核心监控“摘要可用率”（用户未大改就使用的比例）。当发现这个指标偏低时，我的归因方法是：先看用户行为数据（比如哪些部分总被编辑），再抽样分析具体bad case。当时发现，问题集中在“遗漏关键数据指标”上。这就把“不好用”的模糊反馈，精准定位到了“对数字和关键实体识别不足”这个算法问题上。随后，我们将“提升对特定类型信息的提取完整率”作为一个明确的优化需求推动迭代，并在下一版中看到了核心指标的显著提升。

**Mentor：基于你的实践，你认为AI产品经理和传统产品经理最根本的不同在哪里？**
> 我：最根本的不同在于，AI产品经理需要管理“不确定性”，并围绕它来设计“确定性”的体验。传统功能中，一个按钮的响应是确定的；但AI功能的每次输出都可能存在波动。这导致工作重心变化。例如，在设计一个对话功能时，我和设计师争论的焦点往往不是界面本身，而是：当AI回答不完美时，我们应该提供“重新生成”、“局部编辑”还是“让AI自我修正”作为核心纠错路径？这要求我必须对模型的能力边界、常见的错误类型以及迭代速度有预判，才能制定正确的体验策略。所以，我的核心工作之一就是设计一套人机协作的交互协议，让用户能高效地与一个不完美的智能体共事。

**Mentor：理解。如果让你把熟悉的某个功能，从“辅助工具”升级为能自主工作的“智能体（Agent）”，你会怎么设计？考虑哪些风险？**
> 我：以“周报生成”为例，升级为Agent意味着它需要具备感知、规划、执行的能力。我的设计是：它能在设定时间自动触发，主动去调取我在日历、项目管理系统和代码库中的活动记录，自主判断工作亮点与难点，生成结构化草稿。这里的关键是，它需要调用多个工具，并拥有一个简单的“价值判断模块”。同时，我会设计三重护栏（Guardrail）：一是操作确认，任何涉及发送或分享的举动都必须经我同意；二是过程可解释，生成的草稿能关联到信息源；三是结果可控，提供便捷的修正入口。本质是让Agent从“被动工具”变为“主动的初级助手”，但控制权必须牢牢掌握在用户手中。

**Mentor：这个设计涉及多系统协作。如果让你规划一个“跨团队信息同步Agent”，你认为最大的复杂性在哪里？**
> 我：最大的复杂性在于信息的异构性和权限的碎片化。不同团队用的工具、文档格式乃至沟通“黑话”都不同，Agent需要理解“一次核心代码提交”比“十次日常沟通”更重要。更大的挑战是权限，企业内信息孤岛众多。我的思路可能是采用一种“联邦式”架构：为各团队或系统设计轻量适配器，先在本域内完成信息梳理与脱敏，再由一个中心协调器进行汇总，而不是试图打造一个能通行无阻的“超级Agent”。我会向技术团队强调，这个项目的核心验收标准不是“信息抓取量”，而是“为管理者节省的关键信息甄别时间”。

**Mentor：我们团队强调“AI Native”工作方式。你在过往的产品工作里，是否用AI重塑过自己的工作流程？**
> 我：有的，我系统地将大模型用在了竞品分析上。传统方式是人工海量阅读，效率低。现在，我会先收集信息，然后通过一系列指令让大模型扮演不同角色：例如，先作为“用户”提炼痛点，再作为“分析师”梳理竞品迭代路径，最后作为“顾问”输出机会点分析。我的角色就从信息收集员，转变为策略的提问者、框架的设计者和AI输出的审阅者。这节省了大量基础工作时间，让我能更聚焦于深度思考和判断。我相信，未来的产品经理，核心能力在于提出对的问题和做出对的决策。

**Mentor：（追问）战略层面，像智谱这样的大模型公司，做办公Agent，对比拥有海量场景的互联网公司，优劣势是什么？我们该采取什么策略？**
> 我：智谱的核心优势是“模型原生”，能在Agent最核心的“大脑”（如复杂推理、规划）层面进行深度、灵活的优化，技术迭代路径短。相对的挑战是缺乏原生的、高频的垂直场景和数据。因此，我认为策略不应该是复制一个通用的办公套件，而应采取“引擎+生态”的模式。一方面，选择几个高价值场景（如项目复盘），打造体验极致的标杆应用，证明技术价值。另一方面，将Agent的核心能力（如规划引擎、工具调用框架）封装成最好的“发动机”和“开发工具包”，开放给合作伙伴，让他们去构建丰富的场景化应用。这样既能发挥我们的技术长板，又能快速融入更广阔的生态。

**Mentor：最后一个问题，近期有哪个AI技术进展让你特别兴奋，并思考过其产品化可能？**
> 我：我对智能体（Agent）框架的进展，特别是能展示“思考链”并动态规划行动的技术很感兴趣。它让AI不仅给出答案，还能展示推理过程，这极大地提升了可解释性和可控性。例如，在处理一个复杂用户查询时，Agent可以展示出“我需要先查A，再结合B，最后验证C”的步骤。这不仅能增加用户信任，也让调试和优化变得更有针对性。我通过阅读论文、体验开源框架来跟进，并会思考如何将这种“透明化思考”的能力转化为提升产品体验的具体功能点。

3. 结论先行，再展开说。不要一来说一大堆，最后说答案，会让人很没有耐心。

我的指南：2026年构建AI代理需要使用哪些工具（如果你是新手）
讨论

每个人都是从零开始的。如果你是人工智能开发新手，并且被铺天盖地的“十大人工智能代理框架”文章搞得晕头转向（简直一团糟），那么我推荐你试试我日常使用的框架。我相信它不仅对新手来说最简单易用，而且也是最具可扩展性、通用性和生产就绪性的。
我以构建人工智能工具和开源项目为生，我已经批量删除了足够多的失败实验，因此知道哪些有效，哪些无效！
所以，以下是我对2026年的建议（但再过一个月，谁知道呢……）：
1. 听我说……如果你现在只想找个能用的代理，那就选 OpenClaw。
如果你不想从零开始构建，只想立即运行一个程序，OpenClaw 就是你的最佳选择。它在 GitHub 上拥有超过 6 万个星标，支持自托管，可以连接到 Telegram/WhatsApp，具备内存管理、任务调度功能，以及一个完整的工具市场。只需输入你的 API 密钥，连接一些服务，就大功告成了，你的代理程序就能真正执行各种任务。
工具生态系统才是真正的吸引力所在。你可以连接搜索、邮件、数据库、支付等等。就搜索而言，Brave 在二月份取消了免费 API 服务，这让很多依赖它进行开发的用户损失惨重。我转而使用 Valyu，注册即可获得免费额度，搜索结果质量非常高，可以直接替代 Brave，而且它还有 OpenClaw 的技能支持。（它还具备深度研究功能，我用它来处理繁重的研究任务。）
现在说句实话：如果你不知道什么是命令行界面（CLI），那就别自己部署 OpenClaw。 我是认真的。微软安全团队专门发布了一篇博文，讲解如何安全运行它。已经出现过远程代码执行（RCE）漏洞的案例，市场上也充斥着一些不靠谱的技能，还有人反映他们的代理程序陷入循环，一夜之间就浪费了价值数百美元的 API 额度。这软件本身其实不错，但像这样爆红的开源项目的问题在于，很多人没有仔细阅读安装说明，结果就犯了各种愚蠢的错误。
2. 如果你想构建自己的应用，可以使用 Vercel AI SDK + Next.js。
如果您想构建定制化的功能，而不是配置现成的解决方案，那么 Vercel AI SDK 就是您的最佳选择。它能处理 99% 的繁琐样板代码。 useChat 只需大约 15 行代码，它就能为您提供一个可用的流媒体聊天界面。
但真正重要的是：它与服务提供商无关。只需编写一次代码，即可在 Claude、OpenAI、Gemini 等任何服务商之间切换，无需重写应用程序。在价格几乎每周都在变动的情况下，这一点至关重要。
将其与 Next.js 结合使用，即可在一个代码库中实现流媒体、服务器操作、API 路由、身份验证和前端，并在 30 秒内部署到 Vercel。
我并非有意为 Vercel 做广告，但他们的生态系统确实最容易上手，尤其适合新手。而且根据我的经验，它也最容易扩展到大型生产应用。
3. 使用 OpenAI / Claude 构建模型
这两个服务商都不错。例如，GPT-5-mini 价格非常便宜，而且足以应对大多数情况。Claude Opus 在处理更长的上下文和更严谨的推理方面表现出色。
一个小技巧：很多人不知道的是，OpenAI 有一个数据共享计划，你可以选择加入，允许他们使用你的 API 流量进行训练，作为回报，你每天都能获得免费代币。主模型每天最多可以获得 100 万个代币。前往“设置”→“组织”→“数据控制”→“共享”即可找到。当然，如果你要处理任何敏感数据，就不要开启这个功能。但对于业余项目和实验来说，免费代币就是免费的，哈哈。他们已经多次延长了该计划，所以记得查看一下它是否仍然有效。
4. 工具使用方面的 MCP 或技能
MCP（模型上下文协议）由 Anthropic 引入，OpenAI 和 Google 现在也采用了它。本质上，它们是连接器，可以让你的智能体与外部服务通信，而无需你为所有内容编写自定义 API 封装。这是我们目前最接近标准的方案。但最近，技能（解释如何使用服务的 Markdown 文件……）变得越来越流行。在大多数情况下，使用 MCP 还是技能都无关紧要，但是：
我真正会首先选择的是：
Supabase—— 这个代理程序可以直接读写你的数据库。看到它运行起来真是有点不可思议。
Valyu——允许您的代理人搜索网络以及实时财务数据等信息。
Stripe  - 来自代理内部的支付
PostHog  - 直接从代理发出的分析查询
Context7—— 这个功能被低估了。它能从实际的源代码仓库中实时拉取特定版本的文档到你的命令行提示符中。再也不用看到 Claude 自信满满地对着一个六个月前就已弃用的 API 写代码了。
Gmail  - 阅读和发送电子邮件
modelcontextprotocol.io 上的注册表现在有数百个了。六个月前只有大约十二个。Vercel 也有一个技能库，地址是 skills(.)md。
5. 使用光标或 Claude Code 实际编写代码
你不必事事亲力亲为。Cursor 是一款 AI 代码编辑器，Claude Code 也能在终端里完成类似的工作。你只需告诉它们中的任何一个“使用 Vercel AI SDK 为我构建一个智能体，使其能够利用这些 MCP 执行 X 操作”，一个小时后就能运行起来。我可不是开玩笑。现在唯一的瓶颈在于你是否能够清晰地表达你想要实现的目标。

评论区：把 Claude 的代码集成到了 GitHub CLI 中（基本上就是 Claude 的 GitHub CLI 使用技能）。这极大地提升了我的工作流程，我可以让它创建 PR、合并 PR、创建分支、创建 issue 等等，而无需离开终端。一个小小的改动却带来了巨大的生活便利

关于 MCP 部分，有一点值得补充：最大的突破不在于单个 MCP，而在于它们的组合。当一个代理能够在一个请求中同时查询 Supabase、Gmail 和 Stripe 时，它 就从“AI 助手”升级到了“真正能够闭环的 AI”。大多数代理演示都止步于跨工具整合这一步，而这正是其真正价值的起点。
