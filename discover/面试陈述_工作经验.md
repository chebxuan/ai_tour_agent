## 一、一句话定位（电梯演讲）

> 我在一家入境游创业公司，从 0 到 1 搭建了覆盖 10 个城市的模块化产品体系和供应商网络，并独立开发了一套规则引擎系统来自动化产品匹配和报价计算——从手动拉 Excel 到输入城市/天数/人数一键生成中英双语的行程单和报价表。

## 二、工作全景：我做了什么（按层次讲）

### 第一层：业务基建 — 产品和供应商体系搭建

**背景：** 公司面向海外英文用户提供定制化中国文化旅行，需要可销售的产品和可用的供应商。

**我做了什么：**

1. **定义产品架构** — 采用"宜家式模块化"思路，将每个城市的景点/导游/酒店/车辆定义为"最小服务单元"，再组合成半日游 → 一日游 → 多日游产品包。这个架构的好处是：成本可追溯、定价透明、组合灵活。

2. **搭建产品库（10 城市 × 21 产品）** — 从主要城市 2 日经典线路开始，延伸至 3/4 日深度游，覆盖北京、上海、西安、重庆、阳朔、张家界、广州、贵州、云南、成都。每个产品包含每日行程、常规/可选项目、项目编号、成本明细。

3. **搭建供应商库** — 在小红书和微信社群中寻找和对接当地地接社、个人导游、酒店资源。关键做法：**事前准备好所有需要的信息清单，一次性沟通到位**，不做零散追问。这让供应商觉得我们是专业的、值得长期合作的。

4. **产出客户交付物** — 全英文 Canva 产品菜单（Product Menu + Product Introduction），包含产品目录、行程介绍、Highlights、价格参考。还搭建了城市交通数据库和图库作为配套资产。

5. **处理真实询单** — 为一个客户定制了 10 天中国行程，对接地接社/导游/酒店/车辆，输出路书、行程单和每日提醒，做全程线上支持。

**体现的能力：** 业务建模、流程标准化、跨文化沟通（全英文交付物）、供应商管理、客户服务

### 第二层：自动化系统 — 从手工到规则引擎

**痛点驱动：** 每次报价要手工拉一遍 Excel 明细表，繁琐易错，且产品库、成本库、报价表三者之间没有自动化关联。

**我做了什么：**

1. **数据架构设计（四个层次）**：
   - 产品定义层：`product_library.csv`（21 个产品真源）
   - 服务目录层：按类型组织的 5 个价格 CSV（门票/交通/导游/酒店/推荐）
   - 成本运行时层：按城市合并的 `mashes/*.csv`（10 个城市，供引擎直接消费）
   - 展示增强层：`products.normalized.json`（结构化 day_plans）+ `attraction_highlights.json`（景点 Highlights 映射）
   
   ⚠️ **面试可讲的亮点：** 这个四层架构的本质是"源数据与运行时分离开"，数据变更只需改源 CSV → 跑脚本 normalize → 系统自动生效。这是后端开发中 data pipeline 的思维方式。

2. **规则引擎系统（Python FastAPI）**：
   - `product_engine.py`：城市 + 天数精确匹配，CSV 查表，21 产品全覆盖
   - `cost_engine.py`：按产品项目编号 × 人数构成 × 淡旺季 × 儿童/老人折扣计算总价，支持门票/酒店/交通/导游四大费用类别的分类汇总和人均报价
   - `city_config.py`：城市配置中心（代码前缀、成本库路径、折扣规则、车型自动选择）
   - `payment_tracker.py`：供应商付款管理（三列 Kanban：待处理→已支付→已存档），支持多条件筛选和状态流转

3. **前端展示层**：
   - Streamlit Web UI：运营人员可在浏览器直接填写需求，一键生成行程和报价
   - CLI 工具：交互式问卷 → 产品匹配 → 报价计算
   - `/api/v2/full_chain`：一键全链路接口，输入城市/天数/人数即返回 Markdown 行程单

**体现的能力：** 系统设计、数据建模、Python 后端开发、API 设计、前端原型

### 第三层：迭代与决策 — 尝试 → 验证 → 删减

这部分在面试中非常有说服力，因为它展示了你不是"只会写代码"，而是会做技术决策。

1. **Coze（扣子）平台探索 → 放弃**
   - 尝试用 Coze 做 AI Agent 对话式旅游规划，完成了 Bot 创建、OpenAPI 插件配置、多轮对话 Prompt 设计
   - 发现平台对复杂 Schema 支持不完善，且与业务需求匹配度不高
   - **主动决定放弃，移除全部 Coze 相关代码**，避免维护两套系统

2. **V1 API → 精简到 V2**
   - V1 有 7 个端点（含 Coze 集成和飞书预留接口），随着需求明确，发现大量冗余
   - 删除全部 V1 端点、Coze 集成、飞书桩代码，仅保留 V2 核心接口

3. **删除 Plan/Delivery Engine — 减少中间层**
   - 最初设计了 5 段管道：Product → Pricing → Plan → Delivery → Markdown
   - 后发现 Plan Engine 只是把 JSON 重新打包成 Pydantic 对象，Delivery Engine 输出的全是模板/占位符文字
   - **决定删除两个 engine**，让 `full_chain` 直接从 `products.normalized.json` 的 day_plans + `attraction_highlights.json` 的 Highlights 生成 Markdown
   - 效果：`api_main.py` 从 1553 行减至 948 行（-39%），`schemas.py` 从 460 行减至 268 行（-42%），删除了 20 个废弃 Pydantic 模型

**体现的能力：** 技术决策、代码重构、YAGNI 原则（You Aren't Gonna Need It）、工程判断力

⚠️ **面试时可强调：** "我不是舍不得删代码的人。写代码是为了解决问题，不是给自己攒遗产。中间层如果没有增量价值，就该删掉。"

## 三、STAR 面试故事（准备 3 个）

### 故事 1：从手工到自动化 — "为什么要写这个系统"

- **S（情境）：** 公司有 21 个产品 × 10 个城市，每次客户询价都要手工打开 Excel，算门票、酒店、交通、导游四类费用，再根据人数/淡旺季调整，容易出错且效率低。
- **T（任务）：** 我决定做一个系统：输入城市/天数/人数，自动匹配产品并输出报价。
- **A（行动）：**
  - 先梳理数据：把所有产品的 item codes 与成本库 CSV 建立映射关系，修复了 31 组映射错配
  - 用 Python FastAPI 写了 product_engine（匹配）和 cost_engine（报价），支持儿童/老人折扣、淡旺季差价、车型自动选择
  - 写了 Streamlit 前端让运营同事直接在浏览器操作
- **R（结果）：** 10 个城市全部通过端到端回归测试。报价从"手工拉表"变成"填参数 → 出结果"。
  ⚠️ **你需要补充：** "原来手工一次报价要多久？现在多久？团队有多少人用？"

---

### 故事 2：独立搭建产品和供应商体系

- **S（情境）：** 公司起步阶段，面向海外英文用户的入境游产品从零开始。
- **T（任务）：** 我需要定义产品架构、搭建产品库、寻找供应商、制作客户交付物。
- **A（行动）：**
  - 设计"模块化产品"思路：最小服务单元 → 半日/一日游 → 多日产品包，每个层级成本可追溯
  - 在小红书/微信社群寻找和对接供应商，**事前准备好全部问题清单一次性沟通**，提高效率
  - 用 Canva 制作全英文产品菜单，包含行程介绍、Highlights、价格参考
  - 处理真实客户询单，输出 10 天定制行程路书
- **R（结果）：** 建成了 10 城市 × 21 产品的产品矩阵，配套供应商网络、成本明细、全英文交付物料。
  ⚠️ **你需要补充：** "对接了多少家供应商？产品上线后有没有实际成单？客户反馈如何？"

---

### 故事 3：删代码比写代码更需要判断力

- **S（情境）：** 系统中存在 Coze 集成、V1 API、Plan Engine、Delivery Engine 等大量代码，但实际只用 V2 全链路。
- **T（任务）：** 简化系统，减少维护负担。
- **A（行动）：**
  - 分析各模块的实际数据流，发现 Plan/Delivery Engine 只是把 JSON 重新打包，没有增量价值
  - 删除 2 个 engine 文件、2 个 API 端点、20 个废弃 Pydantic 模型
  - 重写 Markdown 生成函数，直接从数据源读取 day_plans 和 Highlights
  - 代码量减少 ~40%，且不影响任何功能
- **R（结果）：** 系统从 5 段管道简化为 3 段（Product → Pricing → Markdown），路由从 21 个精减到 19 个，全部 V2。
  ⚠️ **你可补充：** "如果是你，面对一个已经能跑但冗余的系统，你会怎么做？我选择的是..."

## 四、你的差异化优势（面试中主动提）

### 1. 业务 + 技术 双线能力
你不是纯工程师，也不是纯运营。你**理解业务痛点 → 设计方案 → 自己动手写代码实现**。这是创业公司和中小团队最需要的能力——不需要一个 PM 传话给工程师，你可以在自己脑子里完成这个对话。

### 2. 全英文业务环境
产品菜单全部英文撰写，面向海外用户，与海外客户直接沟通。这在"AI + 出海"赛道上是一个实打实的加分项。

### 3. 从 0 到 1 的完整经历
不是加入一个已有体系的团队做螺丝钉，而是**自己定义产品架构、自己搭建数据体系、自己找供应商、自己写系统**。这种经历在面试中讲故事时，完整度和说服力远超"我在大厂做了某个模块的某个功能"。

### 4. 系统思维而非功能堆砌
四层数据架构、引擎分层、管道设计、废弃代码清理——这些不是"我会写 Python"能概括的，而是**你在思考系统怎么设计才算合理**。

### 5. 行业 Know-how 不可替代
入境游是一个有壁垒的垂直领域：中国供应商生态、景点票务体系、多城市交通衔接、英文导游市场——这些知识不是看两篇文章就能获取的。
